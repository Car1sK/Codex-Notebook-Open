notebooklm-py 与 Open Notebook 适配分析

状态：首个双向迁移闭环已实现并通过定向测试。当前范围是显式 bundle 迁移与直接笔记管理，不包含后台自动同步。

当前已实现

  • notebooklm-py 提供版本化 bundle 导出与发布，覆盖 notebook、source 和 note。
  • Open Notebook 提供预览、幂等导入和本地 notebook bundle 导出三个 REST 入口。
  • external_sync_mapping 保存远端实体到本地实体的映射和内容指纹，重复导入按 create/update/skip 处理。
  • Codex 可通过独立 MCP 服务使用六个 Open Notebook notebook/note 工具，不依赖 Google 登录。
  • 本地启动脚本负责数据库、worker、API、前端和加密环境变量。

已确认的最佳集成 seam

Open Notebook 集成模块 → notebooklm-py 类型化客户端 → Google NotebookLM

不共享数据库，不让 Open Notebook 直接理解 Google 的位置数组 RPC，也不让 notebooklm-py 依赖 SurrealDB。Open Notebook 只依赖稳定的类型化接口；所有 Google 协议变化继续由 notebooklm-py 隔离。

双方职责

系统保持所有权新增职责
notebooklm-pyGoogle 认证、RPC、远端 Notebook/Source/Note/Artifact提供可复用的 Notebook 导出快照或批量读取接口
Open Notebook本地 Source/Note、向量、模型、Podcast、SurrealDB提供 NotebookLM connector、映射、同步任务和 REST/UI 入口

建议的模块形状

  • notebooklm-py：公开不可变 NotebookBundle、SourceBundle 和 export_bundle()，集中并发读取 source fulltext、notes 和可选 artifact 元数据。
  • Open Notebook/open_notebook/integrations/notebooklm.py：bundle 校验、预览、幂等导入、导出和映射存储。
  • Open Notebook/api/routers/integrations.py：bundle 预览、导入和导出。
  • SurrealDB 新增外部映射记录：provider、profile、remote type/id、local id、fingerprint、last synced time。
  • 认证只引用本机 notebooklm-py profile；不把 Google Cookie 存入 Open Notebook 数据库。

不能直接假设的事项

  • “双向同步”不等于完整镜像；双方能力不对称。
  • NotebookLM text source 没有稳定内容级去重键，失败重试可能产生副本。
  • 远端更新冲突需要明确策略：只新增、远端覆盖、本地覆盖或人工确认。
  • Google Cookie/Profile 是账号级凭据，只适合本机或受控单租户部署。
  • Open Notebook 的 source 处理与 embedding 可异步执行，导入完成不能只看记录已创建。

后续扩展顺序

  1. 按实际需求增加 artifact 下载映射和可查询的同步状态。
  2. 在真实数据上验证大文件、失败重试和 source embedding 的完成语义。
  3. 只有明确需要时才增加增量同步和冲突策略；默认不做后台自动双向覆盖。

当前完成的 tracer bullet：导出 bundle → 预览 create/update/skip → 导入 → 重复运行不重复创建 → 可从 Open Notebook 导出同一共享契约。工作区不使用 GitHub Issues。
